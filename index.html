5.1 — Purpose of STP

Why STP Exists

Redundant paths in a switched Ethernet network create Layer 2 loops, causing:

  • MAC address table instability — constant changes from looping broadcasts
  • Link saturation — frames loop endlessly consuming all bandwidth
  • High CPU utilization on switches and end-devices
  • Broadcast storms — abnormally high broadcasts that can disable a network within seconds

Unlike IPv4 (TTL) or IPv6 (Hop Limit), Ethernet has no built-in mechanism to limit looping frames. STP was invented specifically to solve this.

Spanning Tree Protocol (STP) — IEEE 802.1D

Invented by Radia Perlman at Digital Equipment Corporation (1985). The Spanning Tree Algorithm (STA) creates a loop-free topology by:

  • Selecting a single root bridge as the reference point for all switches
  • Intentionally blocking redundant paths (placing ports in blocking state)
  • Recalculating and unblocking when a link fails — providing redundancy

Spanning tree is enabled by default on Cisco switches.

A Host Caught in a Layer 2 Loop

Is completely inaccessible to other hosts. The switch cannot reliably forward unicast frames because the MAC address table keeps changing with every looping broadcast frame — an unknown unicast is flooded out all ports except the ingress, making the loop even worse.

5.2 — STP Operations

4 Steps to a Loop-Free Topology (STA)

  • Step 1 — Elect the root bridge: Switches exchange BPDUs; the switch with the lowest BID becomes root
  • Step 2 — Elect root ports: Every non-root switch selects 1 port with the lowest cost path to root
  • Step 3 — Elect designated ports: One designated port per segment — the port with the best path to root on that segment
  • Step 4 — Elect alternate (blocked) ports: All remaining ports go to blocking state to prevent loops

Switches use BPDUs (Bridge Protocol Data Units) — sent every 2 seconds — to exchange information and make all these decisions.

Bridge ID (BID) Structure

BID = Bridge Priority (2 bytes) + Extended System ID / VLAN (12 bits) + MAC Address (6 bytes)

  • Default bridge priority: 32768 (range: 0–61440 in increments of 4096)
  • Lower BID = preferred root bridge. Priority 0 takes precedence over all others.
  • Tie-break: same priority → lowest MAC address (in hex) wins
  • Note: Default BID = 32769 (32768 priority + 1 for VLAN 1)

Root Path Cost — Port Costs

The internal root path cost = sum of all port costs along the path to the root bridge. Lower cost = preferred path.

Root Port Tie-Breaking (Multiple Equal-Cost Paths)

When a switch has multiple equal-cost paths to the root bridge:

  • 1st: Lowest sender BID
  • 2nd: Lowest sender port priority (default: 128)
  • 3rd: Lowest sender port ID

All tiebreakers are based on the sender's values, not the receiver's.

STP Timers

  • Hello Timer: 2 sec default (range 1–10) — interval between BPDUs
  • Forward Delay: 15 sec default (range 4–30) — time spent in each of Listening and Learning states
  • Max Age: 20 sec default (range 6–40) — max wait before attempting a topology change

Default timers are set on the root bridge and apply to the entire STP domain.

Total delay from Blocking → Forwarding (standard STP): 30 seconds (15 sec Listening + 15 sec Learning).

Per-VLAN Spanning Tree (PVST)

A separate STP instance runs per VLAN. Different root bridges can be elected for different sets of VLANs, enabling load balancing. If all ports are in VLAN 1, only one spanning tree instance exists.

5.3 — Evolution of STP

STP Versions — Full Comparison

VersionDescription
STP (CST)Original IEEE 802.1D. One instance for the entire bridged network regardless of VLANs. Also called Common Spanning Tree.
PVST+Cisco enhancement — separate 802.1D instance per VLAN. Supports PortFast, UplinkFast, BackboneFast, BPDU guard, BPDU filter, root guard, loop guard. Default on Cisco IOS 15.0+.
802.1D-2004Updated STP incorporating IEEE 802.1w (RSTP).
RSTP (802.1w)Rapid Spanning Tree Protocol — faster convergence (sometimes milliseconds). Uses 3 port states. Backward compatible with STP.
Rapid PVST+Cisco's RSTP + PVST+ = separate 802.1w instance per VLAN. Requires explicit configuration.
MSTP / MSTMaps multiple VLANs to same spanning tree instance. MST = Cisco implementation, up to 16 RSTP instances.

RSTP Port States vs STP

RSTP reduces 5 STP states to only 3 states. The Disabled, Blocking, and Listening states are merged into a single Discarding state.

RSTP port roles: Root, Designated, Alternate (replaces blocked — has alternate path to root), Backup (backup to shared medium like a hub).

RSTP can achieve convergence in as little as a few hundred milliseconds vs. STP's 30–50 seconds.

PortFast

Allows a switch access port to bypass the 30-second Listening + Learning delay — immediately transitions to Forwarding. Prevents DHCP clients from timing out during STP convergence.

  • Use only on access ports connecting end devices (PCs, printers, IP phones)
  • Never use on switch-to-switch links — this would create a spanning tree loop risk

BPDU Guard

When enabled on a PortFast port, immediately puts the port into errdisabled state if any BPDU is received. A PortFast port should never see BPDUs — that means a switch was connected where only an end device should be.

The administrator must manually re-enable the port after an errdisabled event.

Why Move Away from STP?

Layer 3 routing allows redundant paths and loops in topology without blocking ports. Routing protocols offer better convergence predictability. Some enterprise designs now use Layer 3 everywhere except the access layer — connections between access and distribution switches are Layer 3 links, eliminating the need for STP at those layers.

6.1 — EtherChannel Operation

What is EtherChannel?

A link aggregation technology that groups multiple physical Ethernet links into one single logical link. It provides:

  • Fault-tolerance — loss of one physical link doesn't change the logical topology
  • Load sharing — traffic is distributed across all physical links
  • Increased bandwidth — combined speed of all member links
  • Redundancy — without STP blocking the bundle

When configured, the resulting virtual interface is called a port channel. STP sees EtherChannel as a single logical link — if STP blocks it, the entire bundle is blocked (not individual ports).

Implementation Restrictions

  • Cannot mix interface types (e.g., FastEthernet + GigabitEthernet in same group)
  • Maximum 8 ports per EtherChannel group
  • Max bandwidth: 800 Mbps (Fast) or 8 Gbps (Gigabit)
  • Cisco Catalyst 2960 supports up to 6 EtherChannels
  • All member ports must be Layer 2
  • Physical port configuration must be consistent on both devices
  • Config on port-channel interface affects all physical members

PAgP — Port Aggregation Protocol (Cisco Proprietary)

Sends packets every 30 seconds to manage EtherChannel links. Checks configuration consistency.

6.2 — Configure EtherChannel

Configuration Guidelines

  • All interfaces must support EtherChannel (don't need to be physically contiguous)
  • Same speed and duplex on all member interfaces
  • All interfaces must be in the same VLAN or all configured as trunks
  • Same allowed VLAN range on all trunk interfaces
  • Ports with different native VLANs cannot form an EtherChannel

LACP Configuration — 3 Steps

  • Step 1: interface range f0/1 - 2 — select the physical interfaces
  • Step 2: channel-group 1 mode active — create the port channel using LACP
  • Step 3: interface port-channel 1 → then switchport mode trunk + switchport trunk allowed vlan 10,20

For PAgP: use mode desirable or mode auto. For no negotiation: mode on (both sides must use on).

Important: When modifying an existing EtherChannel, remove and re-add the port-channel interface rather than changing directly — to avoid STP errdisabled state.

6.3 — Verify & Troubleshoot EtherChannel

Verification Commands

  • show interfaces port-channel — general status of the port-channel interface
  • show etherchannel summary — one line per port channel; shows if up/down
  • show etherchannel port-channel — info on a specific port channel
  • show interfaces etherchannel — role of each physical member interface
  • show run | begin interface port-channel — view detailed port-channel config

Common EtherChannel Issues

  • Ports not in the same VLAN or not all configured as trunks
  • Ports with different native VLANs — cannot form EtherChannel
  • Trunking configured on only some ports in the bundle, not all
  • Different allowed VLAN ranges — won't form even if PAgP is set to auto/desirable
  • Incompatible negotiation modes — e.g., one side is PAgP auto, other is LACP active
  • Speed or duplex mismatch between member interfaces

7.1 — DHCPv4 Concepts

How DHCPv4 Works

DHCPv4 (Dynamic Host Configuration Protocol v4) dynamically assigns IPv4 addresses and configuration. The server leases an address for a limited period (typically 24 hours to 1 week+).

  • When lease expires, client must request another — usually gets the same address back
  • Lease mechanism prevents powered-off devices from hoarding addresses
  • A Cisco router can serve as a DHCPv4 server (no dedicated server required)
  • DHCPv4 service is enabled by default on Cisco IOS

DORA — 4-Step Lease Process

  • D — DHCPDISCOVER: Client broadcasts to find a DHCP server (source: 0.0.0.0, dest: 255.255.255.255)
  • O — DHCPOFFER: Server offers an available IP address
  • R — DHCPREQUEST: Client broadcasts acceptance of the offer (also notifies other DHCP servers that their offers were rejected)
  • A — DHCPACK: Server acknowledges and confirms the lease

2-Step Lease Renewal Process

  • DHCPREQUEST: Client sends directly (unicast) to the original DHCP server before lease expires. If no response, client broadcasts DHCPREQUEST to other servers.
  • DHCPACK: Server verifies lease info and confirms renewal

Note: DHCPOFFER and DHCPACK can be sent as unicast or broadcast per RFC 2131.

7.2 — Configure a Cisco IOS DHCPv4 Server

3-Step Configuration

  • Step 1 — Exclude addresses:
    ip dhcp excluded-address 192.168.10.1 192.168.10.10
    Reserve static addresses for routers, servers, printers
  • Step 2 — Create pool:
    ip dhcp pool LAN-POOL-1
    Enters Router(dhcp-config)# mode
  • Step 3 — Configure pool options (in dhcp-config mode):
    • network 192.168.10.0 255.255.255.0
    • default-router 192.168.10.1
    • dns-server 8.8.8.8 8.8.4.4
    • domain-name example.com
    • lease 7 (days) or lease infinite

Verification Commands

  • show running-config | section dhcp — view all DHCP config
  • show ip dhcp binding — IP-to-MAC address bindings issued
  • show ip dhcp server statistics — count of messages sent/received

Disable: no service dhcp  |  Re-enable: service dhcp

DHCPv4 Relay Agent

When the DHCP server is on a different subnet from clients, routers don't forward broadcasts by default. Solution:

ip helper-address 192.168.11.6 (configure on the interface facing the clients)

This converts the DHCP broadcast from the client into a unicast sent to the DHCP server.

7.3 — Configure a DHCPv4 Client

Router as DHCPv4 Client

Cisco routers in SOHO/branch environments can act as DHCP clients (e.g., connecting to an ISP via DSL or cable modem).

ip address dhcp (on the WAN interface — interface config mode)

Verify: show ip interface g0/0/1 — confirms interface is up and address was allocated.

Home routers are typically preconfigured as "Automatic Configuration – DHCP" on the WAN side, connecting to the ISP as a DHCPv4 client.

8.1 — IPv6 GUA Assignment

Three RA Message Flags

ICMPv6 Router Advertisement (RA) messages contain 3 flags that tell hosts how to obtain a GUA:

  • A flag Address Autoconfiguration — A=1: use SLAAC to create a GUA from the advertised prefix
  • O flag Other Configuration — O=1: get additional info (DNS, domain) from a stateless DHCPv6 server
  • M flag Managed Address Configuration — M=1: use a stateful DHCPv6 server for all IPv6 configuration
AOMMethod Used
100SLAAC only (default when ipv6 unicast-routing is enabled)
110SLAAC for GUA + stateless DHCPv6 for extra config
001Stateful DHCPv6 — server provides everything

IPv6 link-local address (fe80::/10) is automatically created when the host boots and the Ethernet interface is active — even without a router present.

8.2 — SLAAC

SLAAC Overview

Stateless Address Autoconfiguration — hosts create their own IPv6 GUA without any DHCPv6 server. It's "stateless" because no server maintains address binding records.

  • Router sends RA every 200 seconds
  • Host sends Router Solicitation (RS) to ff02::2 to request an RA immediately (doesn't wait 200 sec)
  • Router joins all-routers group via ipv6 unicast-routing — begins sending RA to ff02::1
  • Default: A=1, O=0, M=0 — SLAAC only

Interface ID Generation Methods (last 64 bits)

Host gets the 64-bit prefix from RA, then generates the 64-bit Interface ID using either:

  • Random generation — Windows 10 default. Better privacy (MAC not exposed).
  • EUI-64 — Takes 48-bit MAC address, inserts hex fffe in the middle → 64-bit ID. Privacy concern: MAC address is embedded in the IPv6 address.

Windows, Linux, and Mac OS all allow configuring which method to use.

Duplicate Address Detection (DAD)

Before using a self-generated IPv6 address, the host runs DAD:

  1. Host sends ICMPv6 Neighbor Solicitation (NS) to a specially constructed solicited-node multicast address (uses last 24 bits of its IPv6 address)
  2. If no Neighbor Advertisement (NA) reply is received → address is unique → safe to use
  3. If an NA is received → address is a duplicate → must generate a new interface ID and repeat DAD

IETF recommends DAD even though the chance of a 64-bit duplicate is extremely remote. Most OSes perform DAD on all IPv6 unicast addresses.

8.3 — DHCPv6

DHCPv6 Operation — 6-Step Process

  1. Host sends RS (Router Solicitation)
  2. Router responds with RA (indicating to use DHCPv6)
  3. Host sends DHCPv6 SOLICIT to find a DHCPv6 server
  4. Server responds with ADVERTISE
  5. Host responds to server (REQUEST or INFORMATION-REQUEST)
  6. Server sends REPLY

UDP ports: client → server = 547  |  server → client = 546

8.4 — Configure DHCPv6 Server

DHCPv6 Router Roles

  • DHCPv6 Server — provides stateless or stateful DHCPv6 services to clients
  • DHCPv6 Client — acquires IPv6 configuration from a DHCPv6 server
  • DHCPv6 Relay Agent — forwards DHCPv6 messages between different networks

Stateless DHCPv6 Server — 5 Steps

  1. ipv6 unicast-routing — enable IPv6 routing
  2. ipv6 dhcp pool POOL-NAME — define pool
  3. Configure pool: dns-server, domain-name
  4. ipv6 dhcp server POOL-NAME — bind pool to interface
  5. ipv6 nd other-config-flag — set O flag = 1 (RA tells clients to get extra info from DHCPv6)

Stateful DHCPv6 Server — 5+ Steps

  1. ipv6 unicast-routing
  2. ipv6 dhcp pool POOL-NAME
  3. Configure: address prefix 2001:db8:acad:1::/64, dns-server, domain-name
  4. ipv6 dhcp server POOL-NAME
  5. ipv6 nd managed-config-flag — set M flag = 1
  6. ipv6 nd prefix default no-autoconfig — set A flag = 0 (don't use SLAAC)

DHCPv6 Relay Agent

When client and DHCPv6 server are on different networks, configure relay on the interface facing clients:

ipv6 dhcp relay destination 2001:db8::1 g0/0/0

The egress interface is only required when the next-hop address is a Link-Local Address (LLA).

Verify: show ipv6 dhcp interface and show ipv6 dhcp binding
